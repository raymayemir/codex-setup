# Starting From Scratch

Use this reference when the task is a new surface, a broad redesign, or a vague "make it better" request.

## Start With The Feature

- Begin with the core feature or task, not the app shell.
- List the information, inputs, outputs, and decisions the user needs for that task.
- Only design surrounding chrome after the core feature has shape.
- If the shell is not clarified by the feature, keep it temporary and lightweight.

## Work In Tight Cycles

Move in this order:
1. Core flow and content
2. Grouping and hierarchy
3. Layout and spacing
4. Type and color systems
5. Depth, imagery, and finishing polish

Do not obsess over icons, shadows, gradients, or micro-details while the flow is still unstable.

## Delay Decoration Until The Structure Works

- If the interface still feels confusing in grayscale, more color will not save it.
- If the grouping is unclear, more shadows and borders will usually make it worse.
- Make the surface work with simple blocks, neutral colors, and basic type before adding flourish.

## Choose Personality On Purpose

Pick the personality across a few explicit axes:
- Tone: sober, friendly, premium, editorial, playful, technical, utilitarian
- Typography: neutral sans, warm humanist sans, refined serif pairing, bold display accent
- Corners: square, restrained radius, or soft rounded
- Density: compact, balanced, or spacious
- Copy voice: formal, direct, conversational, or energetic

Match the level of expression to the context:
- Sober product UI: internal tools, settings, enterprise dashboards, admin flows
- Polished product UI with character: most SaaS surfaces, customer dashboards, onboarding flows
- Expressive marketing/editorial UI: launches, campaigns, branded storytelling, premium showcases

## Limit Choices Early

Define constrained systems instead of improvising every decision:
- spacing scale
- type scale
- font weights
- radii
- elevation levels
- color ramps

Good design is easier when the available choices are few and visibly different from each other.

## Useful Outputs To State Explicitly

Before coding, be able to summarize:
- the primary user task
- the single most important action
- the intended personality
- the density target
- the design system tokens you plan to reuse or define

If you cannot state these clearly, keep designing before polishing.
